This project estimates monthly residential burglary risk in London and converts predicted hotspots into a patrol planning problem. The point was not to automate policing decisions, but to build a decision-support tool that makes demand visible.

Report-level result: 82.96% accuracy and 0.8716 recall for burglary cells.

Task

The project question was: how can police demand be estimated in an automated manner so resources can be allocated more effectively for residential burglary prevention?

Data

The core input was Police.uk street-level crime data. Since locations are anonymized to snap points, exact address interpretation would be misleading. The pipeline therefore aggregates incidents into 500m grid cells and monthly labels.

Feature groupExamples
Crime historyPrevious-month burglary, three-month mean, neighbor activity.
WeatherTemperature, rainfall, daylight duration.
SocioeconomicHouseholds, deprivation score, mean house price, young adult share.
SpatialLSOA, ward, 500m grid cell, residential land-use fraction.

Model

Early tabular regression attempts were useful as baselines, but they did not preserve the spatial structure needed for patrol planning. The final approach used a ConvLSTM: convolution handles local spatial patterns and the LSTM state keeps track of change over recent months.

Allocation

A hotspot map is not yet a patrol plan. Predicted high-risk cells were intersected with residential land-use areas, assigned to wards, converted to estimated patrol time, and passed to an optimization model with officer-hour constraints. 

predicted risk cells
  -> keep residential area
  -> estimate patrol time
  -> ward-level officer constraints
  -> patrol schedule / CSV output

Limitations

The model should be treated as advisory. It inherits reporting bias from historical crime data, lacks hourly timing, uses anonymized locations, and does not model full travel time. This is exactly why the output should support human planning rather than replace it.
